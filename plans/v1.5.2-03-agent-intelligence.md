# Feature Implementation Plan: Dynamic Agent Intelligence (v1.5.2-03)

## 🔍 Analysis & Context
*   **Objective:** Give the AI agent richer, computed athlete context by adding a `get_athlete_status` tool (W/kg, weight in lbs, CTL/ATL/TSB) and injecting those live metrics into the system prompt alongside the existing structured benchmarks.
*   **Affected Files:**
    - `server/coaching/tools.py` (New `get_athlete_status` tool)
    - `server/coaching/agent.py` (Enhanced system prompt with computed metrics)
    - `tests/test_coaching_tools.py` (Test new tool)
    - `tests/test_coaching_api.py` (Test enhanced prompt)
*   **Key Dependencies:** PostgreSQL (psycopg2), Google ADK (for tool registration).
*   **Risks/Edge Cases:**
    - **Missing `daily_metrics`:** First-time users or un-synced athletes might have no `daily_metrics` row. The tool and prompt must gracefully handle `None` values for CTL/ATL/TSB.
    - **Division by zero:** W/kg calculation must handle zero or missing weight_kg.

### What already exists (no changes needed)
- `athlete_settings` table with `is_active` versioning — full setting history is already retained (no separate `athlete_log` table needed).
- `update_athlete_setting` tool in `planning_tools.py` — already calls `set_athlete_setting` correctly.
- `AGENTS.md` — contains no hardcoded personal metrics, only glossary definitions.
- `DEFAULT_ATHLETE_PROFILE` — qualitative text (experience, goals, constraints), no hardcoded metrics.
- Structured benchmarks (FTP, weight, HR, age, gender) are already injected into the system prompt via `get_all_athlete_settings()`.

## 📝 Step-by-Step Implementation Details

#### Phase 1: `get_athlete_status` Tool
1.  **Step 1.A (Test):** Define `get_athlete_status` expectations.
    *   *Target File:* `tests/test_coaching_tools.py`
    *   *Test Cases:*
        - Assert `get_athlete_status()` returns all expected keys: `weight_kg`, `weight_lbs`, `ftp`, `w_kg`, `lthr`, `max_hr`, `resting_hr`, `age`, `gender`, `ctl`, `atl`, `tsb`, `as_of_date`.
        - Assert kg-to-lbs conversion is correct (multiply by 2.20462).
        - Assert W/kg = ftp / weight_kg.
        - Assert graceful handling when `daily_metrics` has no rows (CTL/ATL/TSB return `None`).
2.  **Step 1.B (Implementation):** Create `get_athlete_status() -> dict` in `server/coaching/tools.py`:
    - Pull `get_all_athlete_settings()` for raw settings.
    - Pull `get_current_pmc_row(conn)` for latest CTL/ATL/TSB.
    - Convert `weight_kg` to `weight_lbs` (× 2.20462).
    - Calculate `w_kg` = `ftp / weight_kg` (guard against zero).
    - Return combined dictionary.
3.  **Step 1.C (Registration):** Add `get_athlete_status` to the `tools` list in `_get_agent()`.

#### Phase 2: Enhanced System Prompt
1.  **Step 2.A (Test):** Validate enhanced system prompt.
    *   *Target File:* `tests/test_coaching_api.py`
    *   *Test Cases:* Mock `get_all_athlete_settings()` and `get_current_pmc_row()`, then call `_build_system_instruction(None)` and assert the output contains W/kg, weight in lbs, and CTL/ATL/TSB values.
2.  **Step 2.B (Implementation):** Enhance `_build_system_instruction` in `server/coaching/agent.py`:
    - After fetching `benchmarks`, also fetch the latest PMC row via `get_current_pmc_row`.
    - Add computed fields to the benchmarks section: `Weight (lbs)`, `W/kg`, `CTL`, `ATL`, `TSB`, `Metrics as-of date`.
    - Handle missing PMC data gracefully (show "N/A" or "No data").

#### Phase 3: Contextual Analysis & Planning Feedback
1.  **Step 3.A (Test):** Write test for `get_planned_workout_for_ride`.
    *   *Target File:* `tests/test_coaching_tools.py`
    *   *Test Cases:* Assert tool returns the planned workout for a ride date, and returns `None`/empty gracefully when no planned workout exists.
2.  **Step 3.B (Implementation):** Implement `get_planned_workout_for_ride(ride_id: str) -> dict` in `server/coaching/tools.py`:
    - Look up the ride's date, then find any planned workout scheduled for that date.
    - Return planned workout details (type, target TSS, intervals, etc.) alongside the ride's actual summary for comparison.
    - Register in `agent.py`.
3.  **Step 3.C (Implementation):** Update the agent's analysis behavior:
    - When analyzing a ride, if a planned workout exists for that date, include a "Planned vs Actual" comparison in the system prompt or tool guidance.
    - Flag significant deviations (e.g., TSS much higher/lower than planned, wrong zone focus).
4.  **Step 3.D (Implementation):** Implement agent-suggested "Update Week" logic:
    - When deviations are detected (missed workouts, over/under-training), the agent should proactively suggest adjustments to the remaining week using existing `replace_workout` tooling.

### 🧪 Testing Strategy
*   **Unit Tests:** Verify math (W/kg, lb/kg conversions) handles division by zero and zero/missing weights. Verify PMC data is `None`-safe. Verify planned workout lookup handles missing data.
*   **Integration Tests:** Start an agent session and ask "What is my current W/kg, FTP, and TSB?" to assert it uses the injected prompt context instead of hallucinating.
