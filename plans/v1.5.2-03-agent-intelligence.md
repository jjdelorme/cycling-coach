# Feature Implementation Plan: Dynamic Agent Intelligence (v1.5.2-03)

## 🔍 Analysis & Context
*   **Objective:** Integrate live physiological and training metrics from `athlete_settings`, `daily_metrics`, and `athlete_log` into the AI agent's prompt context and tooling, ensuring atomic updates and removing stale hardcoded values.
*   **Affected Files:**
    - `server/database.py` (Helper functions for atomicity, defaults)
    - `server/coaching/tools.py` (New `get_athlete_status` tool)
    - `server/coaching/planning_tools.py` (`update_athlete_setting` implementation)
    - `server/coaching/agent.py` (System prompt injection)
    - `AGENTS.md` (Cleanup of hardcoded values, update docs)
    - `tests/test_coaching_tools.py` (Test new tool)
    - `tests/test_planning_tools.py` (Test write-through logic)
*   **Key Dependencies:** `sqlite3` (via psycopg2 style API in local `get_db()`), Google ADK (for tool registration).
*   **Risks/Edge Cases:**
    - **Type conversion:** `athlete_settings` stores values as `TEXT`, but `athlete_log.value` is `REAL`. Safe casting is required when writing to the log.
    - **Missing `daily_metrics`:** First-time users or un-synced athletes might have no `daily_metrics` row. The prompt and tools must gracefully handle `None` values for CTL/ATL/TSB.
    - **Context Management:** Legacy users who previously customized `coach_settings.athlete_profile` will still have the hardcoded values instead of placeholders. We need to document that they should incorporate placeholders (like `{{FTP}}`) into their custom prompts to regain dynamic functionality, while the defaults get updated immediately.

## 📝 Step-by-Step Implementation Details

### Prerequisites
*   Ensure the local database matches the expected schema (`athlete_log` and `athlete_settings` exist).

#### Phase 1: Database Write-Through & Atomicity
1.  **Step 1.A (The Unit Test Harness):** Define the verification requirement for atomic updates.
    *   *Target File:* `tests/test_database.py`
    *   *Test Cases to Write:*
        - Assert `set_athlete_setting_with_log("ftp", "280", "Agent override")` correctly updates `athlete_settings` and inserts a new row in `athlete_log`.
        - Assert that invalid float conversions (e.g., value="unknown") gracefully skip logging but still update the text setting without crashing the transaction.
2.  **Step 1.B (The Implementation):** Create atomic DB operation.
    *   *Target File:* `server/database.py`
    *   *Exact Change:* Add `set_athlete_setting_with_log(key: str, value: str, note: str = "")` helper that executes both queries within a single `with get_db() as conn:` block. Cast `value` to `float` for `athlete_log.value`, setting it to `None` if it fails to parse. Use `datetime.now().isoformat()` for the `date`.
3.  **Step 1.C (The Implementation):** Connect tool to new atomic logic.
    *   *Target File:* `server/coaching/planning_tools.py`
    *   *Exact Change:* Modify `update_athlete_setting` tool to call `set_athlete_setting_with_log` instead of `set_athlete_setting`. Pass `note="Updated via AI Coach agent"`.

#### Phase 2: Agent Tool Specification
1.  **Step 2.A (The Unit Test Harness):** Define `get_athlete_status` expectations.
    *   *Target File:* `tests/test_coaching_tools.py`
    *   *Test Cases to Write:* Assert `get_athlete_status()` returns exactly the required keys, properly converting kg to lbs and calculating W/kg.
2.  **Step 2.B (The Implementation):** Implement new read tool.
    *   *Target File:* `server/coaching/tools.py`
    *   *Exact Change:* Create `get_athlete_status() -> dict`:
        - Pull `get_all_athlete_settings()` (source: `athlete_settings` table).
        - Pull `get_current_pmc_row(conn)` (source: latest `daily_metrics` table entry).
        - Convert `weight_kg` to `weight_lbs` (multiply by 2.20462).
        - Calculate `w_kg` = `ftp / weight_kg`.
        - Return dictionary with keys: `weight_kg`, `weight_lbs`, `ftp`, `w_kg`, `lthr`, `max_hr`, `resting_hr`, `ctl`, `atl`, `tsb`, `as_of_date`.
    *   *Target File:* `server/coaching/agent.py`
    *   *Exact Change:* Import and add `get_athlete_status` to the `tools` list in `_get_agent()`.

#### Phase 3: Dynamic Prompt Injection
1.  **Step 3.A (The Unit Test Harness):** Validate system prompt parsing.
    *   *Target File:* `tests/test_coaching_api.py`
    *   *Test Cases to Write:* Mock DB settings to contain `{{FTP}}` and ensure the agent runner's system prompt resolves the integer correctly.
2.  **Step 3.B (The Implementation):** Prepare default contexts.
    *   *Target File:* `server/database.py`
    *   *Exact Change:* Modify `DEFAULT_ATHLETE_PROFILE` to replace the literal text with placeholders:
      ```text
      - {{AGE}}-year-old {{GENDER}}, ~{{WEIGHT_LBS}} lbs ({{WEIGHT_KG}} kg)
      - Current FTP: {{FTP}}w, W/kg: {{W_KG}}
      - Current Fatigue State: CTL {{CTL}}, ATL {{ATL}}, TSB {{TSB}} (As of {{AS_OF_DATE}})
      - LTHR: {{LTHR}} bpm, Max HR: {{MAX_HR}} bpm
      ```
3.  **Step 3.C (The Implementation):** Inject the context.
    *   *Target File:* `server/coaching/agent.py`
    *   *Exact Change:* Inside `_build_system_instruction(ctx)`, right after fetching settings:
      - Call `get_athlete_status()` to get live numbers.
      - Call `get_all_athlete_settings()` to get string traits like Age/Gender.
      - Perform `.replace()` across `settings['athlete_profile']`, `settings['coaching_principles']`, `settings['coach_role']`, and `settings['plan_management']` for all supported placeholder keys.
4.  **Step 3.D (The Cleanup):** Document and sanitize.
    *   *Target File:* `AGENTS.md`
    *   *Exact Change:* Remove mentions of hardcoded "163 lbs" or specific FTP numbers to prevent confusion. Add a section explaining Dynamic Prompt Variables (e.g., `{{FTP}}`) so future developers understand the injection mechanism.

### 🧪 Global Testing Strategy
*   **Unit Tests:** Verify math (W/kg, lb/kg conversions) handles division by zero and zero weights correctly. Verify atomic transactions in the DB helper gracefully handle string-to-float failure without rolling back the text setting update.
*   **Integration Tests:** Start an agent session and ask "What is my current W/kg, FTP, and TSB?" to assert it uses the injected prompt context instead of hallucinating. Verify updating a setting logs correctly to `athlete_log`.

## 🎯 Success Criteria
*   [ ] `get_athlete_status` is available to the agent and returns standard metric schemas calculating W/kg perfectly based on current values.
*   [ ] Calling `update_athlete_setting` writes exactly one row to `athlete_settings` and one row to `athlete_log` within the same transaction.
*   [ ] System prompt is generated on-the-fly, replacing `{{FTP}}`, `{{WEIGHT_KG}}`, etc., with live database data.
*   [ ] `AGENTS.md` and database defaults contain no hardcoded personal metrics, relying entirely on the stateful database configuration.
