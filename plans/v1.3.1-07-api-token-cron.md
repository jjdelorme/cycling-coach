# Plan: API Token for Automated Sync (Cron)

**Version:** 1.3.1
**Type:** Feature
**Complexity:** Low
**Approach:** Simple CLI token minting (no schema changes)

## Summary

Add a CLI command to mint a long-lived JWT for cron-based sync automation via curl. Same JWT verification path as browser tokens, with `"type": "api"` claim for auditability.

## Current State

- Auth uses Google Sign-In + app JWT (24h expiry via `JWT_EXPIRY_HOURS`)
- `create_app_token()` produces JWTs: email, name, picture, exp, iat, iss
- `verify_app_token()` validates HS256, issuer="cycling-coach", checks expiry
- `get_current_user()` decodes JWT, extracts claims, calls `_upsert_user()` for role
- Sync endpoint requires `require_write` (readwrite or admin role)
- `server/__main__.py` only runs ingestion
- No API key or service account mechanism exists

## 1. `create_api_token()` in `server/auth.py`

Add below `create_app_token()`:

```python
def create_api_token(email: str, name: str = "", expiry_days: int = 365) -> str:
    """Create a long-lived API token for automated/cron use."""
    now = datetime.datetime.now(datetime.timezone.utc)
    payload = {
        "email": email,
        "name": name,
        "picture": "",
        "type": "api",
        "exp": now + datetime.timedelta(days=expiry_days),
        "iat": now,
        "iss": "cycling-coach",
    }
    return jwt.encode(payload, JWT_SECRET, algorithm="HS256")
```

- `picture=""` for consistency with `get_current_user()` which reads `claims.get("picture", "")`
- `type: "api"` is informational — `verify_app_token()` doesn't check it
- Same issuer ensures `verify_app_token()` accepts it

## 2. Verification Path — No Changes Needed

`get_current_user()` handles API tokens without modification:
1. Reads `Authorization: Bearer <token>`
2. `verify_app_token()` succeeds (same secret, algorithm, issuer)
3. Extracts email/name/picture
4. `_upsert_user()` returns role
5. Role guards work normally

The `type: "api"` claim is ignored by `jwt.decode()`.

## 3. CLI Entry Point — Rewrite `server/__main__.py`

Replace with argparse CLI supporting `ingest` and `mint-token`:

```python
"""CLI entry point for server management commands.

Usage:
    python -m server ingest
    python -m server mint-token --email you@gmail.com [--expiry-days 365]
"""

import argparse, sys
from server.database import get_db, init_db

def cmd_ingest(args):
    from server.ingest import run_ingestion
    run_ingestion()

def cmd_mint_token(args):
    from server.config import JWT_SECRET
    from server.auth import create_api_token
    if not JWT_SECRET:
        print("Error: JWT_SECRET not set.", file=sys.stderr)
        sys.exit(1)
    init_db()
    with get_db() as conn:
        row = conn.execute("SELECT email, display_name, role FROM users WHERE email = %s", (args.email,)).fetchone()
    if not row:
        print(f"Error: No user '{args.email}'. Must log in via web UI first.", file=sys.stderr)
        sys.exit(1)
    if row["role"] not in ("readwrite", "admin"):
        print(f"Error: User role '{row['role']}' insufficient. Need readwrite or admin.", file=sys.stderr)
        sys.exit(1)
    token = create_api_token(email=row["email"], name=row["display_name"] or "", expiry_days=args.expiry_days)
    print(f"Token minted for {row['email']} (role: {row['role']}), expires in {args.expiry_days} days.", file=sys.stderr)
    print(token)  # bare token to stdout for capture

def main():
    parser = argparse.ArgumentParser(prog="python -m server")
    sub = parser.add_subparsers(dest="command", required=True)
    sub.add_parser("ingest")
    mint = sub.add_parser("mint-token")
    mint.add_argument("--email", required=True)
    mint.add_argument("--expiry-days", type=int, default=365)
    args = parser.parse_args()
    {"ingest": cmd_ingest, "mint-token": cmd_mint_token}[args.command](args)

if __name__ == "__main__":
    main()
```

- Token to stdout, metadata to stderr — `TOKEN=$(python -m server mint-token ...)` captures only JWT
- Role validation warns if user can't call sync endpoint
- User must exist (logged in via web UI at least once)
- `python -m server ingest` still works

## 4. Cron Setup

```bash
# Mint
TOKEN=$(python -m server mint-token --email you@gmail.com --expiry-days 365)
echo "SYNC_TOKEN=$TOKEN" > ~/.config/coach/sync-token.env
chmod 600 ~/.config/coach/sync-token.env

# Crontab (every 6 hours)
0 */6 * * * . $HOME/.config/coach/sync-token.env && curl -sf -X POST https://your-host/api/sync/start -H "Authorization: Bearer $SYNC_TOKEN" >> /var/log/coach-sync.log 2>&1
```

Token renewal: re-mint before expiry. Old token continues working until its exp.

- Cannot be revoked without rotating JWT_SECRET (acceptable for single-user)
- `_upsert_user()` updates `last_login` on each cron call (audit signal)
- `type: "api"` enables future filtering
