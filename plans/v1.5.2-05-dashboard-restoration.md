# Phase 5: Dashboard & UI Restoration (v1.5.2)

**Goal:** Restore the "Actual vs Planned" visual context to the dashboard by reconciling historical ride data with future training intent.

## Technical Research Summary
- **Backend**: The `GET /api/rides/summary/weekly` endpoint currently only aggregates from the `rides` table and does not account for future weeks or planned targets.
- **Frontend**: `Dashboard.tsx` uses `chart.js` for the "Weekly Training Load" bar chart but lacks the infrastructure to display overlapping datasets (Actual vs. Planned).
- **Data Models**: The `planned_workouts` table contains `planned_tss` and `total_duration_s`, which are the primary targets for comparison.

## Implementation Plan

### 1. Unified Weekly Aggregation API
Update `GET /api/rides/summary/weekly` in `server/routers/rides.py` to act as a unified training load engine.

- **Window Logic**: Default to a rolling 12-week window.
  - History: Previous 8 ISO weeks.
  - Current: The current ISO week.
  - Future: The next 3 ISO weeks.
- **Aggregation Algorithm**:
  1. Calculate the start and end dates for the 12-week window using `datetime.isocalendar()`.
  2. **Query A (Actuals)**: Fetch all `rides` between `start_date` and `end_date`.
  3. **Query B (Plans)**: Fetch all `planned_workouts` between `start_date` and `end_date`.
  4. **Merge Logic**:
     - Initialize a `defaultdict` or map containing every `YYYY-Wxx` key in the 12-week sequence.
     - Sum `tss` and `duration_s` from rides into their respective ISO week buckets.
     - Sum `planned_tss` and `total_duration_s` from planned workouts into the same buckets.
  5. **Normalization**: Return a sorted list of `WeeklySummary` objects, ensuring zero-values for weeks with no activity to maintain chart continuity.
- **Schema Update**: Add `planned_tss` (float) and `planned_duration_h` (float) to the `WeeklySummary` Pydantic model in `server/models/schemas.py`.

### 2. The "Ghost Bar" Visualization
Modify the "Weekly Training Load" chart in `frontend/src/pages/Dashboard.tsx`.

- **Library**: Continue using `react-chartjs-2` with `chart.js`.
- **Chart Configuration**:
  - **Dataset A (Background/Ghost)**: `planned_tss` using a low-alpha color (e.g., `rgba(233, 69, 96, 0.2)`). These "Ghost Bars" represent the target.
  - **Dataset B (Foreground/Solid)**: `actual_tss` using a high-alpha color (e.g., `rgba(233, 69, 96, 0.9)`).
  - **Overlay Logic**: Set `scales.x.stacked: false` and adjust `barPercentage` (e.g., Target at 0.8, Actual at 0.5) to make the actual bar appear centered within the ghost bar.
- **Current Week Highlight**: 
  - Identify the current week's index in the dataset.
  - Use a scriptable `backgroundColor` property in Chart.js to apply a "glow" or border (e.g., `cc.accentGlow`) only to the current week's bar.

### 3. UI Polish & Friendly Mapping
Implement consistent, human-readable labels across the dashboard.

- **Sport Name Mapper**: Add `fmtSport(sport: string)` to `frontend/src/lib/format.ts`.
  - Maps `mtb` -> `Mountain Bike`
  - Maps `virtualride` -> `Virtual Ride`
  - Maps `road` -> `Road Cycling`
- **Application**: Update the "Recent Rides" table in `Dashboard.tsx` and the individual ride views to use this formatter.

## Success Criteria
- [ ] Dashboard chart shows a continuous 12-week view (8 past, 1 current, 3 future).
- [ ] Ghost bars clearly indicate training targets vs. actual performance.
- [ ] Current week is visually distinguished from historical and future weeks.
- [ ] Technical strings like "mtb" are replaced with "Mountain Bike" throughout the UI.
