# Feature Implementation Plan: Phase 4 Enhanced Analysis API (v1.5.2)

## 🔍 Analysis & Context
*   **Objective:** Enhance the analysis APIs to include aggregated power curve with HR data, cleaned time-in-zones to ignore power spikes, and enhance the existing efficiency endpoint with rolling averages and endurance filtering.
*   **Affected Files:**
    *   `server/queries.py`
    *   `server/models/schemas.py`
    *   `server/routers/analysis.py`
    *   `tests/test_api.py`
*   **Key Dependencies:** `psycopg2` via `_DbConnection` wrapper (auto-converts `?` and `:name` placeholders).
*   **Risks/Edge Cases:**
    *   Using Postgres-specific `DISTINCT ON` requires ensuring `date::date` casting works with existing `TEXT` dates.
    *   Existing frontend may expect `avg_hr` to optionally not exist, which is handled via schema updates.

## 🔬 Investigation Results (2026-04-01)
The following questions were investigated against the live codebase:

1.  **Placeholder syntax:** Non-issue. `_DbConnection._adapt_sql()` (`server/database.py:270-276`) auto-converts `?` → `%s` and `:name` → `%(name)s`. Both styles work interchangeably.
2.  **Sport column values:** The `rides.sport` column stores Intervals.icu types in **lowercase** (`ride`, `virtualride`, `gravelride`, etc.), NOT `'cycling'`. The cycling check list already exists in multiple places (e.g., `server/services/single_sync.py:94`). The efficiency endpoint must filter against this list, not `sport = 'cycling'`.
3.  **`conn.execute()` pattern:** Works as expected. `_DbConnection` wraps psycopg2 with `RealDictCursor`, providing `conn.execute()` and dict-style row access (`r["column"]`).
4.  **Intervals.icu HR for power bests:** Already handled. `single_sync.py` calls `process_ride_samples()` which passes `hrs` to `compute_rolling_best()` and stores `avg_hr` in `power_bests`.
5.  **`compute_rolling_best` function:** Already updated in Campaign 6. Lives in `server/metrics.py:191`, already accepts `hrs` and `cadences` params, returns `{"power", "avg_hr", "avg_cadence", "start_offset_s"}`. **Phase 2 of the original plan is entirely complete — no work needed.**

## ✅ Already Complete (from prior Campaigns)

    *   *Exact Change:* Replace the current self-join approach in `get_power_bests_rows` with Postgres `DISTINCT ON` to fetch exactly one best per duration with its associated `avg_hr`:
        ```python
        def get_power_bests_rows(conn, start_date: str | None = None, end_date: str | None = None) -> list[dict]:
            if start_date or end_date:
                sd = start_date or "2000-01-01"
                ed = end_date or datetime.now().strftime("%Y-%m-%d")
                rows = conn.execute(
                    """SELECT DISTINCT ON (duration_s) duration_s, power, avg_hr, date, ride_id
                       FROM power_bests
                       WHERE date >= ? AND date <= ?
                       ORDER BY duration_s, power DESC, date DESC""",
                    (sd, ed),
                ).fetchall()
            else:
                rows = conn.execute(
                    """SELECT DISTINCT ON (duration_s) duration_s, power, avg_hr, date, ride_id
                       FROM power_bests
                       ORDER BY duration_s, power DESC, date DESC"""
                ).fetchall()
            return [dict(r) for r in rows]
        ```
4.  **Step 3.D (Update Endpoint):**

#### Phase 5: Enhanced Efficiency Endpoint
> **Note:** An `/efficiency` endpoint already exists at `server/routers/analysis.py:109`. Rather than creating a duplicate `/efficiency-trends` endpoint, this phase enhances the existing endpoint with a 30-day rolling average and endurance ride filtering.

1.  **Step 5.A (The Unit Test Harness):**
    *   *Target File:* `tests/test_api.py`
    *   *Test Cases to Write:* Add/update `test_efficiency` to verify `GET /api/analysis/efficiency` returns HTTP 200 and includes `rolling_ef` in the response. Verify endurance filtering (duration >= 1800s, IF 0.5–0.8) when applicable.
2.  **Step 5.B (Enhance Existing Endpoint):**
    *   *Target File:* `server/routers/analysis.py`
    *   *Exact Change:* Update the existing `/efficiency` endpoint to:
        - Filter for cycling sports using the standard sport list (matching the pattern at `server/services/single_sync.py:94`):
          ```python
          CYCLING_SPORTS = ('ride', 'ebikeride', 'emountainbikeride', 'gravelride',
                            'mountainbikeride', 'trackride', 'velomobile', 'virtualride',
                            'handcycle', 'cycling')
          ```
        - Add endurance ride filtering: `duration_s >= 1800 AND intensity_factor BETWEEN 0.5 AND 0.8`
        - Use Postgres `RANGE BETWEEN INTERVAL '30 days' PRECEDING AND CURRENT ROW` for a rolling EF average
        - Include `rolling_ef` in the response

        The updated query:
        ```python
        query = """
            SELECT id, date, normalized_power, avg_hr, duration_s, sub_sport,
                   (normalized_power::float / avg_hr) as ef,
                   AVG(normalized_power::float / avg_hr) OVER (
                       ORDER BY date::date
                       RANGE BETWEEN INTERVAL '30 days' PRECEDING AND CURRENT ROW
                   ) as rolling_ef
            FROM rides
            WHERE normalized_power > 0 AND avg_hr > 0
              AND sport IN ('ride', 'ebikeride', 'emountainbikeride', 'gravelride',
                            'mountainbikeride', 'trackride', 'velomobile', 'virtualride',
                            'handcycle', 'cycling')
              AND duration_s >= 1800
              AND intensity_factor BETWEEN 0.5 AND 0.8
        """
        ```
        **Note:** The `sport` column (not `sub_sport`) stores the activity type in lowercase. The `WHERE IN (...)` filter replaces the incorrect `sport = 'cycling'` from the original plan.
