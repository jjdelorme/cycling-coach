# Plan: Fix Duplicate Workout Pushes to Garmin

**Version:** 1.3.1
**Type:** Bug
**Complexity:** Medium

## Summary

Three independent paths push workouts to intervals.icu (which syncs to Garmin). The agent tool path has zero deduplication, `push_workout()` always POSTs (creates new), and workout modifications leave stale events.

## Root Cause

- `sync_workouts_to_garmin` in `planning_tools.py` calls `push_workout()` with no dedup
- `push_workout()` always POSTs, never PUTs (updates)
- Modified workouts: old event stays on intervals.icu, new one pushed alongside
- Background sync dedup uses fragile `(date, name)` key that breaks on name changes

## Three Push Paths

1. Agent tool `sync_workouts_to_garmin` — NO dedup
2. Background sync `_upload_workouts` — `(date, name)` dedup (fragile)
3. REST `POST /api/plan/workouts/{id}/sync` — NO dedup

## Phase 2: Idempotent Push

**File:** `server/services/intervals_icu.py`

### 2a. `compute_sync_hash(name, date, zwo_xml, moving_time_secs)` helper

```python
import hashlib
def compute_sync_hash(name, date, zwo_xml, moving_time_secs=0):
    content = f"{date}|{name}|{moving_time_secs}|{zwo_xml}"
    return hashlib.sha256(content.encode()).hexdigest()[:16]
```

### 2b. Modify `push_workout()` — add `icu_event_id` parameter

- If `icu_event_id` provided: `httpx.put()` to `.../events/{icu_event_id}` (update)
- If None: `httpx.post()` to `.../events` (create, current behavior)
- Return `{"status": "success", "event_id": resp.json().get("id")}`

intervals.icu API:
- POST `/api/v1/athlete/{id}/events` — create
- PUT `/api/v1/athlete/{id}/events/{event_id}` — update
- DELETE `/api/v1/athlete/{id}/events/{event_id}` — delete

### 2c. Add `delete_event(event_id)` function

```python
def delete_event(event_id):
    url = f"{BASE_URL}/api/v1/athlete/{athlete_id}/events/{event_id}"
    resp = httpx.delete(url, auth=("API_KEY", api_key), timeout=15.0)
    return {"status": "success"} if resp.status_code in (200, 204) else {"status": "error", ...}
```

# Skip if unchanged

result = push_workout(..., icu_event_id=row["icu_event_id"])  # None=create, int=update

## Phase 4: Fix Background Sync (`_upload_workouts`)

**File:** `server/services/sync.py`

**Remove** the `fetch_calendar_events()` call and `remote_keys` set entirely. Replace with hash-based dedup:

```python
current_hash = compute_sync_hash(name, date, workout_xml, moving_time_secs)
if w.get("sync_hash") == current_hash and w.get("icu_event_id"):
    skipped += 1
    continue
result = push_workout(..., icu_event_id=w.get("icu_event_id"))
# Store event_id and hash on success
```

This also eliminates a potential failure mode where the remote fetch fails and all workouts get re-pushed.

## Phase 5: Stale Event Cleanup

### 5a. `replace_workout` in `planning_tools.py`

Before `DELETE FROM planned_workouts WHERE date = ?`, fetch `icu_event_id`. After transaction commits, call `delete_event(old_event_id)`.

### 5b. `regenerate_phase_workouts` in `planning_tools.py`

Before bulk `DELETE FROM planned_workouts WHERE date >= ? AND date <= ?`, collect all `icu_event_id` values. After transaction, delete each stale event (best-effort, catch exceptions).

### 5c. `generate_weekly_plan` in `planning_tools.py`

Same pattern at `DELETE FROM planned_workouts WHERE date = ?`.

### 5d. `delete_workout` endpoint in `routers/planning.py`

Fetch `icu_event_id` before delete, call `delete_event` after.

### 5e. Per-workout sync endpoint in `routers/planning.py`

Read `icu_event_id`, pass to `push_workout`, store returned event_id and hash.

### Transaction Safety

- All `delete_event()` HTTP calls happen **after** DB transaction commits
- If delete fails, orphaned event on intervals.icu is harmless
- `icu_event_id` defaults to NULL on new rows — no backfill needed
