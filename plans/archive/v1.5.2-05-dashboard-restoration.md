# Phase 5: Dashboard & UI Restoration (v1.5.2)

**Goal:** Restore the "Actual vs Planned" visual context to the dashboard by re-implementing the ghost bar overlay that was removed during the MD3 redesign (commit `425ca91`), and add human-readable sport name formatting across the UI.

## Technical Research Summary

### What Was Lost
The ghost bar visualization was originally added in commit `696468a` and removed as a regression in `425ca91` ("Phase 2: Refine Dashboard with MD3 Tonal Surfaces"). The backend data pipeline is fully intact — only the frontend chart rendering was lost.

### Original Implementation (commit `696468a`)
The original used a **custom Chart.js plugin** (`plannedOverlayPlugin`) that drew dashed, semi-transparent rectangles behind actual bars using the `beforeDatasetsDraw` hook. This required ~400 lines of custom code including:
- Canvas-level drawing with `ctx.fillRect()` / `ctx.strokeRect()`
- Manual bar position calculations via `(bar as any).x` and `(bar as any).width` casts
- Separate scale lookups for dual Y-axes (TSS on `y`, Hours on `y1`)
- Complex data merging of 12 actual weeks + 3 future planned weeks

**Why the plugin approach was problematic:**
- Fragile — relied on Chart.js internal bar element properties via `as any` casts
- ~400 lines of complexity for a visual feature
- Hard to maintain across Chart.js version updates
- The data merging (unified week list with per-bar color arrays) was error-prone

### New Approach: Dataset Ordering (Simpler)
Chart.js supports a `order` property on datasets that controls z-layering. By adding "planned" datasets with `order: 2` (background) behind actual datasets with `order: 1` (foreground), we get the ghost bar effect with **standard Chart.js APIs** — no custom plugin needed.

This reduces implementation from ~400 lines to ~50 lines and eliminates fragile canvas manipulation.

### Current State
- **Backend**: No changes needed. The `planned_workouts` table has `planned_tss` and `total_duration_s`. The `fetchWeekPlan()` API already returns this data.
- **Frontend**: `Dashboard.tsx` already fetches `plannedWeeks` via `fetchWeekPlan()` for the "Next Workout" card. The `planMondays` computation (lines 45-57) and `plannedWeeks` query (lines 60-63) are already in place.
- **API**: `GET /api/plan/week/{date}` endpoint exists in `server/routers/planning.py` (lines 46-65) and returns `{ week_start, week_end, planned[], actual[] }`.
- **Current chart colors**: TSS uses `rgba(233, 69, 96, 0.7)` (theme accent/red), Hours uses `rgba(0, 212, 170, 0.7)` (theme green/teal). Ghost bars will use the **same colors at reduced opacity** to maintain MD3 consistency.

## Implementation Plan

### 1. Restore Ghost Bar Chart (Frontend Only)

Re-implement the weekly volume chart in `Dashboard.tsx` using **dataset ordering** instead of a custom plugin.

#### 1a. Add Helper Functions

The following already exist in `Dashboard.tsx`:
- `planMondays` — already computed (lines 45-57)
- `plannedWeeks` — already fetched (lines 60-63)

**Add:**
- `isoWeekToMonday(isoWeek: string): string` — converts `"2026-W14"` to `"2026-03-30"` for matching actuals to plans.
- `thisMonday` — the current week's Monday, for identifying which week gets the ghost bar treatment.
- `plannedByMonday` useMemo — aggregates `plannedWeeks` into a `Map<string, { tss: number; hours: number }>` keyed by Monday date.

#### 1b. Add Planned Datasets with `order` Layering

Expand the current 2-dataset chart to 4 datasets. The planned datasets render **behind** the actuals via the `order` property:

```typescript
datasets: [
  // Planned bars (order: 2 = rendered behind actuals)
  {
    label: 'Planned TSS',
    data: weekLabels.map(w => {
      const monday = isoWeekToMonday(w)
      return monday === thisMonday ? (plannedByMonday.get(monday)?.tss ?? null) : null
    }),
    backgroundColor: 'rgba(233, 69, 96, 0.15)',    // accent at low alpha
    borderColor: 'rgba(233, 69, 96, 0.35)',
    borderWidth: 1.5,
    borderDash: [4, 4],
    borderRadius: 4,
    yAxisID: 'y',
    order: 2,
  },
  {
    label: 'Planned Hours',
    data: weekLabels.map(w => {
      const monday = isoWeekToMonday(w)
      return monday === thisMonday ? (plannedByMonday.get(monday)?.hours ?? null) : null
    }),
    backgroundColor: 'rgba(0, 212, 170, 0.15)',     // green at low alpha
    borderColor: 'rgba(0, 212, 170, 0.35)',
    borderWidth: 1.5,
    borderDash: [4, 4],
    borderRadius: 4,
    yAxisID: 'y1',
    order: 2,
  },
  // Actual bars (order: 1 = rendered in front)
  {
    label: 'TSS',
    data: weekly.slice(-12).map(w => w.tss),
    backgroundColor: 'rgba(233, 69, 96, 0.7)',
    borderRadius: 4,
    yAxisID: 'y',
    order: 1,
  },
  {
    label: 'Hours',
    data: weekly.slice(-12).map(w => w.duration_h),
    backgroundColor: 'rgba(0, 212, 170, 0.7)',
    borderRadius: 4,
    yAxisID: 'y1',
    order: 1,
  },
]
```

#### 1c. Extended Week List with Future Planned Weeks

Build a unified week list: last 12 actual weeks + up to 3 future planned weeks (only if they have planned data). Future weeks with no actuals show **faded planned-only bars** at 0.25 alpha.

#### 1d. Ghost Bar UX Rules
- Ghost bars appear **only for the current week** when planned workouts exist. No plan = no ghost bar.
- Future weeks with plans but no rides show faded planned-only bars.
- Past weeks show only solid actual bars (no ghost bars for historical comparison).
- Current week shows both: solid actual bar in front, dashed ghost bar behind.

#### 1e. Legend and Tooltip Enhancements
- **Legend filtering**: Hide "Planned TSS" and "Planned Hours" from the legend to avoid clutter. Use `legend.labels.filter` callback.
- **Enhanced tooltips**: For the current week, show "TSS: 280 / 350 planned" format when planned data exists. Use `tooltip.callbacks.label` to merge actual and planned values.
- **Current week highlight**: Bold blue x-axis label for the current week via scriptable `ticks.color` and `ticks.font`.

### 2. Sport Name Mapping

#### 2a. Sport Values in the Database
Sports come from two sources:
- **JSON/FIT imports**: Stored lowercase (e.g., `cycling`, `virtualride`, `mountainbikeride`).
- **Intervals.icu sync**: Uses their taxonomy (e.g., `Ride`, `VirtualRide`, `MountainBikeRide`, `Run`, `Hike`).

The full taxonomy is documented in `AGENTS.md` under "Intervals.icu Activity Taxonomy."

#### 2b. Add `fmtSport()` to `frontend/src/lib/format.ts`
Create a case-insensitive mapper:

| DB Value (case-insensitive) | Display Name |
|---|---|
| `ride`, `cycling` | Ride |
| `virtualride` | Virtual Ride |
| `mountainbikeride` | Mountain Bike |
| `gravelride` | Gravel Ride |
| `ebikeride` | E-Bike Ride |
| `emountainbikeride` | E-Mountain Bike |
| `run` | Run |
| `trailrun` | Trail Run |
| `virtualrun` | Virtual Run |

Handle `null`/`undefined` input by returning a sensible default (e.g., `'Activity'`).

### 3. Frontend Testing Setup & Tests

#### 3a. Add Vitest to Frontend

No frontend test framework currently exists. Set up Vitest (Vite-native, fast):

**Install dependencies:**
```bash
cd frontend && npm install -D vitest
```

**Add to `frontend/package.json` scripts:**
```json
"test": "vitest run",
"test:watch": "vitest"
```

**Create `frontend/vitest.config.ts`:**
```typescript
import { defineConfig } from 'vitest/config'

export default defineConfig({
  test: {
    globals: true,
  },
})
```
