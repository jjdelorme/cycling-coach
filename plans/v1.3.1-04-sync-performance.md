# Plan: Sync Performance Optimization

**Version:** 1.3.1
**Type:** Performance
**Complexity:** Medium

## Summary

Multiple bottlenecks identified in the sync pipeline. The biggest is `executemany` which loops row-by-row instead of batching.

## Bottlenecks Identified

1. **`executemany` is fake** — `database.py:236-240` loops `execute()` per row. A 1-hour ride = 3,600 individual INSERTs.
2. **Serial HTTP fetches** — `sync.py:319-329` fetches stream data one ride at a time.
3. **PMC recomputes from scratch** — `ingest.py:254-293` deletes entire `daily_metrics` table every sync, N+1 weight queries.
4. **Connection churn** — 3+ separate DB connections per ride during sync.

---

## Phase 1: Batch Inserts (Highest Impact, Low Complexity)

**File:** `server/database.py`

Replace fake `executemany` with `psycopg2.extras.execute_batch()`:

```python
def executemany(self, sql, params_list, page_size=1000):
    sql = self._adapt_sql(sql)
    psycopg2.extras.execute_batch(self._cursor, sql, params_list, page_size=page_size)
    return self._cursor
```

- `psycopg2.extras` already imported on line 6
- No SQL format changes needed — `execute_batch` accepts same `%s` placeholders
- `page_size=1000` means up to 1000 statements per round-trip (4 trips for a 1-hour ride vs 3,600)
- Return value: `execute_batch` returns None but no callers use the return value (verified: `sync.py:190`, `ingest.py:363`, `ingest.py:376`)
- **Risk:** LOW. Drop-in replacement. If any row fails, entire batch raises (same as current behavior).

**Expected speedup:** 10-50x for bulk inserts.

## Phase 2: Incremental PMC (Medium Impact, Low Complexity)

**File:** `server/ingest.py`

### Modified signature

```python
def compute_daily_pmc(conn, since_date: str | None = None):
```

### Changes

1. **Prefetch weight data** — single query replacing N+1 pattern:
```python
weight_rows = conn.execute("SELECT date, weight FROM rides WHERE weight IS NOT NULL ORDER BY date").fetchall()
weight_entries = [(r["date"], r["weight"]) for r in weight_rows]
```
Use `bisect.bisect_right` for O(log n) lookup per date.

2. **Carry forward CTL/ATL** when `since_date` provided:
```python
prev_day = (start - timedelta(days=1)).strftime("%Y-%m-%d")
prev = conn.execute("SELECT ctl, atl FROM daily_metrics WHERE date = %s", (prev_day,)).fetchone()
ctl = prev["ctl"] if prev else 0.0
atl = prev["atl"] if prev else 0.0
```
If no previous day found, fall back to full recompute.

3. **Remove DELETE, use UPSERT:**
```sql
INSERT INTO daily_metrics (date, total_tss, ctl, atl, tsb, weight)
VALUES (%s, %s, %s, %s, %s, %s)
ON CONFLICT (date) DO UPDATE SET
total_tss = EXCLUDED.total_tss, ctl = EXCLUDED.ctl,
atl = EXCLUDED.atl, tsb = EXCLUDED.tsb, weight = EXCLUDED.weight
```
Collect all rows and use batch `executemany`.

4. **Determine `since_date` from sync** — modify `_download_rides` to return `tuple[int, int, str | None]` where third element is earliest new ride date.

**Risk:** MEDIUM. CTL/ATL are exponential moving averages — incorrect `since_date` produces wrong values. Mitigation: fall back to full recompute if no previous day exists.

## Phase 3: Connection Reuse (Medium Impact, Low Complexity)

**File:** `server/services/sync.py`

Thread a single `conn` through the sync pipeline:

### Functions needing `conn` parameter

| Function | Change |
|---|---|
| `_store_streams(conn, ride_id, streams)` | Remove `with get_db()`, accept conn |
| `_backfill_start_location(conn, ride_id, streams)` | Remove `with get_db()`, accept conn |
| `get_watermark(key, conn=None)` | Optional conn, backward compat |
| `set_watermark(key, value, conn=None)` | Optional conn, backward compat |
| `_create_sync_run(sync_id, conn=None)` | Optional conn |
| `_update_sync_run(sync_id, conn=None, **kwargs)` | Optional conn |
| `_download_rides(sync_id, log_lines, conn)` | Required conn |
| `_upload_workouts(sync_id, log_lines, conn)` | Required conn |

### Refactored `run_sync`

```python
with get_db() as conn:
    _create_sync_run(sync_id, conn=conn)
    rides_dl, rides_skip, earliest = await _download_rides(sync_id, log_lines, conn)
    conn.commit()
    wo_up, wo_skip = await _upload_workouts(sync_id, log_lines, conn)
    conn.commit()
    if rides_dl > 0:
        compute_daily_pmc(conn, since_date=earliest)
        conn.commit()
    _update_sync_run(sync_id, conn=conn, status="completed", ...)
```

### Transaction boundaries

- Commit after ride download batch, after workout uploads, after PMC recompute
- Failure mid-download rolls back all rides in that sync (re-downloaded next sync via watermark)

**Risk:** LOW-MEDIUM. Long-lived connection during sync (minutes for large sync). Mitigated by existing single-sync guard.

## Phase 4: Parallel Stream Fetching (High Impact, Medium Complexity)

**File:** `server/services/sync.py`

### Restructure

1. Insert all ride metadata (sequential, fast)
2. Fetch all streams in parallel via `asyncio.gather` + `Semaphore(5)`
3. Batch-insert all stream records

```python
async def _fetch_streams_parallel(rides_to_fetch, conn, log_lines, max_concurrent=5):
    sem = asyncio.Semaphore(max_concurrent)
    async def _fetch_one(ride_db_id, icu_id):
        async with sem:
            try:
                streams = await asyncio.to_thread(fetch_activity_streams, icu_id)
                return (ride_db_id, icu_id, streams)
            except Exception as e:
                logger.warning("Could not fetch streams for %s: %s", icu_id, e)
                return (ride_db_id, icu_id, None)
    results = await asyncio.gather(*[_fetch_one(r, i) for r, i in rides_to_fetch])
    # Batch-insert all rows from successful fetches
```

### Extract `_parse_stream_rows` from `_store_streams`

Refactor into pure data transformation (no DB) + DB insertion.

### Error handling

- Each `_fetch_one` catches its own exceptions, returns None
- Failed fetches don't prevent others
- Ride metadata committed regardless

**Risk:** MEDIUM. Parallel HTTP could trigger rate limiting. Semaphore(5) mitigates. Memory for large syncs (~1.3M tuples for 365 rides) is manageable (~100-200MB).
