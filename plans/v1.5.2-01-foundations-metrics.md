# Feature Implementation Plan: Foundations & Metric Integrity (v1.5.2)

## 🔍 Analysis & Context
*   **Objective:** Establish a robust mathematical and data foundation for training metrics (CTL, ATL, TSS, NP) and clarify the "Single Source of Truth" for athlete benchmarks (FTP, Weight, LTHR).
*   **Core Architectural Decisions:**
    1.  **Rides-as-Performance-Truth:** The `rides` table is the immutable record of what happened. Each ride stores the FTP and Weight active at the time of the ride (synced from the source file).
    2.  **Settings-as-Benchmark-Truth:** The `athlete_settings` table is transformed into a **historical log** of benchmarks. This tracks when FTP, Weight, LTHR, etc., were changed and which value is currently "Active" for future planning.
    3.  **Removal of Redundancy:** The `athlete_log` table is deleted (YAGNI).
    4.  **Clean-at-Ingest:** All raw time-series data is cleaned via SciPy (gap filling, spike removal) *before* storage in `ride_records`.
    5.  **Daily State Aggregation:** The `daily_metrics` table is expanded to include `ftp`, providing a complete daily snapshot of the athlete's state (Fitness + Benchmarks).
    6.  **AI Coach Integration:** Structured benchmarks (Age, Weight, FTP, LTHR) are injected directly into the coach's system instruction, while free-text goals remain in the coach's settings.

## 📝 Implementation Details

### Core Benchmarks to Track (athlete_settings)
The following benchmarks are critical for coaching advice and metric calculations:
*   **Primary:** `ftp` (Watts), `weight_kg` (kg), `lthr` (bpm).
*   **Secondary:** `max_hr` (bpm), `resting_hr` (bpm), `age` (yrs), `gender`.

### AI Coach Context Injection
The "ATHLETE PROFILE" in the system instruction will be dynamically built:
```text
ATHLETE PERFORMANCE BENCHMARKS (Structured):
- Current FTP: {ftp}w
- Current Weight: {weight}kg
- LTHR: {lthr}bpm
- Max HR: {max_hr}bpm

ATHLETE PROFILE (Free-text):
{settings['athlete_profile']}
```

### FTP / Weight Source Hierarchy
For any ride calculation, we follow this strict search order:
1. **Source File**: `session.threshold_power` or `user_profile.weight` from the JSON/FIT.
2. **Historical Benchmark**: The value in `athlete_settings` marked `is_active` on the ride's date.
3. **Historical Context**: The value from the most recent previous ride in the `rides` table.
4. **System Default**: `ATHLETE_SETTINGS_DEFAULTS`.

### Signal Cleaning Strategy
*   **Despiking**: Median filtering (3-5s window) handles stochastic anomalies without a hard-coded maximum.
*   **Gap Filling**: Linear interpolation for missing packets (< 10s) ensures continuous rolling averages for NP.

## 🎯 Success Criteria
*   **Consistency**: Every metric (TSS, IF, NP) is calculated using the correct point-in-time benchmark.
*   **AI Intelligence**: The coach uses live, structured benchmarks for planning while retaining context on goals and strengths.
*   **Maintainability**: No redundant tables (`athlete_log` is gone).
*   **Analysis Ready**: `ride_records` contains normalized, high-fidelity data.
