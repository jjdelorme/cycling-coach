# v1.6.3 Analysis & Workout Robustness

## Overview
Address user feedback regarding 0 TSS workouts and UX improvements on the Seasonal Macro-Plan visualization.

## Discovered Root Causes (0 TSS / Missing Targets)
1. **Pydantic Model Stripping**: `PlannedWorkout` schema was missing `planned_tss`, causing FastAPI to strip it from JSON responses even when present in DB.
2. **Generator Strictness**: `_build_zwo_xml` skipped unknown step types (e.g., "Recovery", "Tempo"). If an AI agent used these names, the resulting XML was empty.
3. **Parser Gaps**: `_parse_zwo_steps` only looked for `IntervalsT`, skipping `Intervals` (legacy), `FreeRide`, and `Ramp`.

## 4. Analysis Page Bug Fixes (Post-Release)

Four issues discovered and fixed via automated tester/engineer loop on 2026-04-06.

### 4a. CRITICAL: TDZ Crash on Analysis Page

**Root cause:** In `TrainingPlanOverview`, the `chartData` `useMemo` was defined before `todayIdx` `useMemo`, but referenced `todayIdx` in its callback (for `pointBackgroundColor` and `segment` styling). JavaScript's Temporal Dead Zone caused a `ReferenceError` the moment `overview` data loaded, crashing the entire Analysis page to a blank screen.

**Fix:** Moved `todayIdx` useMemo above `chartData` useMemo in `frontend/src/pages/Analysis.tsx`. Also added `todayIdx` to `chartData`'s dependency array so today-marker colors update correctly when the week boundary changes.

### 4b. WARNING: `weight_kg` Field Mismatch (FTP History Tooltip)

**Root cause:** `get_ftp_history_rows()` in `server/queries.py` aliased the weight column as `weight` in SQL, but returned it in the dict as `"weight"`. The TypeScript type `FTPHistoryPoint` and `Analysis.tsx` both expected `weight_kg`. The field was silently `undefined`, so the Weight line never appeared in FTP chart tooltips.

**Fix:** Changed SQL alias and dict key from `weight` → `weight_kg` in `server/queries.py`.

### 4c. WARNING: `/api/analysis/ftp-history` Ignored Date Range Parameters

**Root cause:** The endpoint accepted no query parameters, so date range selectors (1W / 3M / 6M / 1Y) had no effect on the FTP History tab — all buttons showed identical data.

**Fix:** Added `start_date` and `end_date` as `Optional[str] = Query(None)` to the endpoint in `server/routers/analysis.py` and added conditional `AND date >= / AND date <= ` clauses to `get_ftp_history_rows()` in `server/queries.py`.

### 4d. PERFORMANCE: Slow Zone Distribution Query (~3s)

**Root cause:** The `/api/analysis/zones` query does a full scan of `ride_records` (200K+ per-second rows) joined to `rides`. No composite index exists on `(ride_id, power)` and no index on `rides(date)`, so date-filtered queries always scan the full table.

**Plan:** A migration script has been written at `scripts/migrate_add_zone_indexes.sql`. Run it against the target database to add two idempotent indexes:
- `CREATE INDEX IF NOT EXISTS idx_ride_records_ride_id_power ON ride_records(ride_id, power)`
- `CREATE INDEX IF NOT EXISTS idx_rides_date ON rides(date)`

Expected improvement: sub-500ms for filtered zone queries (from 2.5–3.2s). The migration is safe to run at any time and is fully idempotent.
